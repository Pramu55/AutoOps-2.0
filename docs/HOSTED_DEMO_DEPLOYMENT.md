# Hosted Demo Deployment Guide

This guide explains how to safely deploy AutoOps to a public hosting provider (such as Railway, Render, or Fly.io) for demonstration purposes.

> [!IMPORTANT]
> The hosted demo architecture is designed to be **read-only and isolated** from real infrastructure. It proves the platform UI and workflow capabilities without risking actual deployments.

## Safe Hosted Demo Architecture

AutoOps uses a lazy-loading integration model. When deploying the platform publicly:
1. **Providers Degrade Gracefully:** If you do not provide credentials for AWS, Docker, Jenkins, Kubernetes, or GitHub, the platform does not crash. Instead, those modules return `NOT_CONFIGURED`.
2. **Operations Blocked:** Because there are no credentials, no destructive actions (like deployments or container restarts) can be executed.
3. **No Root or Socket Access:** The hosted demo overlay specifically strips `user: root` and `/var/run/docker.sock` from the API and Worker containers.

By simply deploying the containers and leaving provider credentials blank, you get a fully functional frontend UI with a completely inert and safe backend.

## Why Secrets Are Not Exposed

- **Docker Socket & Kubeconfig:** Removed entirely in `docker-compose.hosted-demo.yml`. The API container physically cannot talk to the host's Docker daemon or a local Kubernetes cluster.
- **Tokens (AWS / Jenkins / GitHub):** Not provided in the `.env.hosted-demo.example` file.
- **Database Segregation:** The seeded database uses the `organizationId` parameter on every query, enforcing tenant isolation.

## Railway Setup (Service-Level)

> [!CAUTION]
> If using another deployment method, ensure PostgreSQL and Redis are **internal-only**. Their ports (5432 and 6379) must NOT be exposed to the public internet. The compose overlay attempts to clear these ports, but hosted platforms (like Railway) handle this via private networking. Always use private/internal database services.

Railway orchestrates multi-container applications as individual connected services. Since AutoOps requires multiple pieces (PostgreSQL, Redis, API, Worker, Web), you will create a Railway project and add them sequentially:

1. **Add Databases:**
   - Add **PostgreSQL** from the Railway templates.
   - Add **Redis** from the Railway templates.

2. **Add the API Service:**
   - Link your GitHub repository.
   - Set the root directory if necessary, and use the `Dockerfile.api`.
   - **Pre-Deploy Command:** `/bin/sh scripts/hosted-demo-init.sh` (Runs migrations & seeds data safely).

3. **Add the Worker Service:**
   - Link the same repository.
   - Use `Dockerfile.worker`.

4. **Add the Web Service:**
   - Link the same repository.
   - Use `Dockerfile.web`.
   - Railway will automatically expose this service to the public via a generated domain.

## Required Environment Variables

You must set the following variables on the Railway services.

> [!TIP]
> Do NOT set the optional provider variables (like `AWS_ACCESS_KEY_ID` or `DOCKER_SOCKET_PATH`). Leave them completely undefined so the integrations remain safely in `NOT_CONFIGURED` state.

### Web Service
- `NODE_ENV`: `production`
- `API_INTERNAL_URL`: URL of the internal API service (e.g., `http://api.railway.internal:4000`)
- `NEXT_PUBLIC_API_URL`: The **public** URL generated by Railway for your API service (used for WebSockets/SSE).

### API & Worker Services
- `DATABASE_URL`: Injected by Railway Postgres.
- `REDIS_URL`: Injected by Railway Redis.
- `NODE_ENV`: `production`
- `STRICT_ENV_VALIDATION`: `false`
- `JWT_SECRET`: Generate a random 64-char string.
- `JWT_REFRESH_SECRET`: Generate a different random 64-char string.
- `SEED_PASSWORD`: The password for the demo users (e.g., `DemoSafePass123!`).
- `PROVIDER_INVENTORY_ALLOWED_ORGANIZATION_SLUGS`: `autoops-demo`
- `CORS_ORIGINS`: The public URL generated for your Web service.

## Local Validation

Before pushing to a hosted platform, you can validate the hosted demo architecture locally using the overlay file:

```powershell
# Verify configuration
docker compose -f docker-compose.yml -f docker-compose.hosted-demo.yml config

# Start the environment
docker compose -f docker-compose.yml -f docker-compose.hosted-demo.yml up -d --build
```

This will run the stack locally, but with all real-infrastructure integrations explicitly disabled, simulating the hosted demo experience exactly.
